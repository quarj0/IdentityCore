# ADR-007: Object Storage for Binary Media

**Status:** Accepted

**Date:** 2026-07-04

---

# Context

IdentityCore processes binary media throughout the identity verification lifecycle.

Examples include:

- Identity Document captures
- Selfie Captures
- Liveness images
- Liveness videos
- Supporting verification evidence
- Future biometric media

These files are typically much larger than traditional database records and require different storage characteristics.

The platform requires a storage solution that is:

- Secure
- Scalable
- Cost-effective
- Highly available
- Independent of the relational database

---

# Decision

IdentityCore will store binary media in **private S3-compatible object storage**.

PostgreSQL will store only metadata and references to media objects.

Binary media must not be stored directly inside PostgreSQL.

---

# Storage Responsibilities

## PostgreSQL

Stores:

- File metadata
- Public IDs
- Object keys
- Content type
- File size
- Checksum
- Upload status
- Processing status
- Retention information

---

## Object Storage

Stores:

- Identity Document images
- Selfie images
- Liveness videos
- Future supporting media

Object storage becomes the authoritative source for binary media.

---

# Object Structure

Example:

```id="yulr0e"
organizations/
    org_01...

        verifications/
            ver_01...

                documents/
                    front.jpg
                    back.jpg

                selfie/
                    selfie.jpg

                liveness/
                    challenge.mp4
```

The physical layout may evolve over time, but logical organization should remain clear and tenant-aware.

---

# Security

All media must be stored in **private buckets**.

Requirements:

- No public object access.
- Encryption at rest.
- Signed upload URLs.
- Signed download URLs.
- Short-lived access tokens.
- Access logging.
- Retention enforcement.

Media must never be permanently exposed through public URLs.

---

# Upload Strategy

Verification Subjects upload media directly to object storage using signed upload URLs generated by the backend.

Flow:

```id="1owwqj"
Verification Portal

↓

Request Upload URL

↓

Django Backend

↓

Signed Upload URL

↓

Object Storage

↓

Upload Complete

↓

Notify Backend
```

This reduces backend bandwidth and improves scalability.

---

# Download Strategy

Downloads should use temporary signed URLs.

Characteristics:

- Time-limited
- Single-purpose where practical
- Generated only after authorization
- Never stored permanently

The backend remains responsible for access control.

---

# Checksums

Every uploaded file should include a checksum.

Checksums support:

- Integrity verification
- Duplicate detection
- Safe processing
- Auditability

Checksum algorithms may evolve over time.

---

# Metadata

Each stored object should have associated metadata such as:

- Object key
- MIME type
- Size
- Upload timestamp
- Checksum
- Processing status
- Tenant
- Verification
- Retention policy

Metadata remains in PostgreSQL.

---

# Versioning

Object versioning is recommended where supported.

Benefits include:

- Recovery from accidental deletion.
- Auditability.
- Safe overwrites.

Versioning policies should balance operational cost and compliance requirements.

---

# Retention

Media retention follows Verification Policies and Compliance requirements.

Examples:

- Delete raw document images after retention period.
- Delete selfie images after retention period.
- Delete temporary uploads after processing.
- Preserve metadata where legally required.

Retention jobs should execute through Celery.

---

# Scalability

Object storage scales independently from PostgreSQL.

Benefits:

- Lower database size.
- Faster backups.
- Better performance.
- Lower operational cost.
- Simplified disaster recovery.

---

# Provider Independence

IdentityCore should remain storage-provider agnostic.

Supported providers may include:

- Amazon S3
- Cloudflare R2
- MinIO
- Enterprise object storage
- Government-managed object storage

Business logic must not depend on provider-specific APIs.

A storage abstraction layer should isolate provider-specific implementations.

---

# Performance

Object storage is optimized for:

- Large files
- High throughput
- Parallel uploads
- Parallel downloads

Database queries remain focused on metadata rather than binary content.

---

# Backup Strategy

Media backups should follow provider capabilities and organizational requirements.

Metadata and media should remain synchronized.

Recovery procedures should verify:

- Object existence
- Metadata integrity
- Checksum consistency

---

# Failure Handling

Upload failures should not create incomplete verification records.

Examples:

- Interrupted upload
- Invalid media
- Storage outage
- Expired upload URL

The backend should:

- Detect incomplete uploads.
- Allow safe retry.
- Clean temporary objects.
- Record failures where appropriate.

---

# Consequences

## Positive

- Better scalability.
- Reduced database size.
- Faster database backups.
- Lower storage costs.
- Provider flexibility.
- Improved upload performance.
- Simplified disaster recovery.

## Negative

- Additional infrastructure.
- Need to synchronize metadata and media.
- More complex upload flow.
- Object lifecycle management required.

These trade-offs are acceptable for a platform handling large volumes of media.

---

# Alternatives Considered

## Store Files in PostgreSQL

Rejected because binary media significantly increases database size, backup time, storage costs, and operational complexity.

PostgreSQL should remain optimized for structured business data.

---

## Local Filesystem

Rejected because local storage:

- Does not scale well.
- Complicates multi-server deployments.
- Increases operational risk.
- Makes disaster recovery more difficult.

Suitable only for local development.

---

## Provider-Specific Storage APIs

Rejected because direct coupling would reduce portability and make migrations more difficult.

An abstraction layer preserves deployment flexibility.

---

# Future Considerations

Future versions may support:

- Cross-region replication.
- Intelligent storage tiers.
- Automatic archival.
- Malware scanning pipelines.
- Image optimization.
- Immutable evidence storage.
- Government-controlled storage backends.

These enhancements should preserve the storage abstraction.

---

# Implementation Notes

- Store media in private object storage.
- Store metadata in PostgreSQL.
- Use signed upload and download URLs.
- Never expose permanent public URLs.
- Apply lifecycle and retention policies through background jobs.
